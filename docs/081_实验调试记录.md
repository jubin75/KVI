# KVI 2.0 Pattern Contract & Evidence-Gated 
一、Retrieval Prompt

## 背景
你正在为 KVI 2.0 设计一个“Pattern-first → Semantic-second → Introspection-gated”的 retrieve pipeline。
当前系统存在以下问题：

1. Pattern、slot、semantic instance 的职责混乱
2. slot 被当作语义对象，而非语义生成的约束接口(注意，这里slot和之前用于拼接prompt的slot不一样)
3. Pattern 与 Evidence 的关系未被工程化表达
4. topic 级配置缺乏明确来源与生成机制

本 Prompt 的目标是：  
**重构 Pattern Contract 体系，使其成为语义生成的前置契约，而非事后校验规则。**

---

## 一、核心设计原则（必须遵守）

### P1. Pattern 不是知识结论
Pattern 只回答一个问题：
> 在什么“可验证的形式条件”下，允许生成某一类语义实例？

Pattern 不得包含具体实体（如某个药物名），只能定义形式约束。

---

### P2. Slot 是语义生成接口，而非知识本体
Slot 的职责：
- 描述一类语义实例“需要哪些字段”
- 标记哪些字段已被 Pattern 满足，哪些仍为空
- 为 introspection 提供可解释失败路径

Slot 不能直接用于回答用户问题。

---

### P3. Semantic Instance 必须 Evidence-Grounded
任何 semantic instance 必须：
- 明确引用 ≥1 个 block_id
- 每个 block_id 必须满足至少一个 Pattern
- 可被 introspection 拒绝或撤回

---

## 二、Topic 级 Pattern Contract 的定义

### 2.1 什么是 Topic
Topic 是一个 **稳定的知识子域**，例如：
- SFTSV
- Alzheimer’s Disease
- CRISPR

Topic 的生命周期 ≥ 单次 query。

---

### 2.2 Topic Pattern Contract 的来源（必须实现）

Topic Pattern Contract **不是人工拍脑袋定义的**，而是通过以下流程生成：

1. 从该 topic 的高质量 evidence block 中抽样
2. 统计高频结构性共现模式（如：
   - 缩写 + 疾病名
   - FDA + drug + investigated
3. 将这些模式固化为 **形式可验证规则**
4. 写入 `topics/{topic}/pattern_contract.json`

---

### 2.3 Pattern Contract 示例（JSON）

```json
{
  "topic": "SFTSV",
  "patterns": {
    "hard": [
      {
        "id": "abbr:SFTSV",
        "type": "lexical",
        "rule": "SFTSV must appear or its full name"
      }
    ],
    "soft": [
      {
        "id": "schema:drug_study",
        "type": "cooccurrence",
        "rule": "drug_name + studied/investigated + disease"
      }
    ]
  }
}

二、Pattern-first Retrieve Pipeline（必须实现）
Step 1：Pattern Retrieval

对 block 做快速稀疏或规则检索

输出：{
  "block_id": "...",
  "matched_patterns": ["abbr:SFTSV", "schema:drug_study"]
}
Step 2：Slot Instantiation

根据 matched_patterns 决定：

哪些 slot 可以尝试填充

哪些 slot 必须保持为空

Slot 示例：{
  "drug": "filled",
  "disease": "filled",
  "mechanism": "empty"
}

Step 3：Semantic Instance Candidate

仅当 slot 满足最低生成条件时

才生成 semantic instance candidate

必须附带 block_id 引用

Step 4：Introspection Gating

Introspection 必须能回答：

哪个 slot 是空的？

是缺 Pattern 还是缺 Evidence？

是否允许 partial answer 或拒答？

三、明确禁止的实现方式
❌ 从 prompt 动态拼 Pattern
❌ 无证据 slot 填充
❌ Pattern = 实体词表

四、需要输出的代码结构

你需要生成以下模块（允许空实现，但接口必须存在）：

PatternContractLoader

PatternMatcher

SlotSchema

SemanticInstanceBuilder

IntrospectionGate

每个模块必须有清晰 docstring，说明其在上述体系中的位置。

最终目标

使 KVI 2.0 具备以下能力：

Pattern 失败 → 可解释拒答

Slot 空缺 → 延迟生成

Evidence 不足 → 明确说明不确定性

Topic 稳定 → 可复现实验

请据此修改或重构现有代码。
