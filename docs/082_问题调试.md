但仍存在 两个架构级隐患：

二、你必须解决的隐患 #1
❗ question_skeleton 缺乏“稳定接口层”
现状（不要删除）
"question_skeleton": ["X 是什么", "X 的全称是什么"]

必须升级为（向后兼容）
"question_skeleton": {
  "intent": "ask_definition",
  "surface_forms": ["X 是什么", "X 的全称是什么"]
}

你的实现要求

Pattern Contract Parsing

支持两种形式：

旧格式：question_skeleton: [str, ...]

新格式：question_skeleton.intent + surface_forms

若为旧格式：

自动提升为：

intent = "legacy"
surface_forms = original_list


PatternMatcher 行为约束

匹配逻辑只允许：

使用 surface_forms 与 prompt 做字符串 / embedding 匹配

intent 不参与匹配

intent 仅作为：

introspection 输出

pattern 稳定 ID 语义标签

IntrospectionGate 输出必须新增

"matched_intent": "ask_definition"


⚠️ 禁止：

使用 LLM 推断 intent

在 runtime 动态生成新 intent

让 prompt 决定 intent

三、你必须解决的隐患 #2
❗ inference_level 尚未约束“允许的回答行为”
当前问题

即使 slot 标记为：

"inference_level": "schema"


系统仍可能：

生成事实性总结

给出超出证据的自然语言断言

这是严重的可控性漏洞。

四、你必须引入的“强约束规则”（不新增模块）
1️⃣ 定义 Answer Capability（内部枚举即可）
AnswerCapability = {
  "FACTUAL_ASSERTION",
  "EXPLANATION",
  "LIST_ONLY",
  "REFUSE"
}

2️⃣ Slot → AnswerCapability 的强制映射规则（必须实现）

在 IntrospectionGate 或等价位置：

If any required slot.inference_level == "schema":
    allowed_capabilities = {LIST_ONLY, EXPLANATION}

If any required slot.inference_level == "soft":
    allowed_capabilities ⊆ {FACTUAL_ASSERTION, EXPLANATION}

If any required slot.inference_level == "hard":
    allowed_capabilities ⊆ {FACTUAL_ASSERTION}

3️⃣ Pattern 级别的最终裁决规则（必须执行）
If allowed_capabilities is empty:
    decision = REFUSE

If PatternSpec.answer_style not in allowed_capabilities:
    decision = REFUSE

4️⃣ IntrospectionGate 必须输出（用于审计）
{
  "allowed_answer_capabilities": ["LIST_ONLY", "EXPLANATION"],
  "final_answer_style": "LIST_ONLY",
  "decision": "ALLOW | REFUSE",
  "decision_reason": "schema-level-slot-limits-answer"
}


⚠️ 禁止：

自动降级 answer_style

LLM 自行“换个说法”

schema slot 触发 factual assertion

五、你必须保证的系统不变量（Invariant）

你生成的代码必须保证以下 永远成立：

Schema slot 永远不能触发事实性回答

Prompt 永远不能改变 Pattern 的 intent

Semantic Instance 永远不生成文本

无 Pattern / 无匹配 / 无合法 answer capability → 直接拒答
