【KVI + IM + SOP + LLM：可审计的临床决策代理（PRD 草案）】

> 系统目标（我们达成共识的定义）：
>
> **“LLM 越强，我们越能把它用在交互与抽取上；但决策与安全边界仍由 SOP+IM 承担，形成可审计的临床决策代理。”**

---

## 1. 背景与问题

医疗场景的核心矛盾：
- **输入不完整**：用户 query 往往缺少关键临床变量（检验、既往史、用药史、严重程度分级等）
- **输出高风险**：一句不严谨的建议会被当作诊疗结论
- **知识更新频繁且有版本**：指南/共识会更新，且有地区/人群差异

仅靠 LLM（哪怕 CoT 更强）会面临：
- 不可控的“看起来合理但不可审计”的推断
- 缺失信息时的自由发挥
- 难回滚、难做严肃的质量保证（QA）与合规审计

---

## 2. 系统目标（Goals）

构建一个“临床决策代理”，使系统在在线推理期能够：
- **可靠决策**：主要决策路径由 SOP/决策树产生（可复现）
- **安全门禁**：IM 在在线期对“是否允许推断/是否允许执行”做强制约束（fail-closed）
- **证据绑定**：KVI 提供证据供给与引用，使关键结论/关键分支可追溯
- **低结构化成本**：对患者状态采用“按需增量结构化”（只收集当前路径必需变量）
- **低维护成本**：指南 PDF → SOP 半自动编译 + 人审 + 版本化回归测试

---

## 3. 非目标（Non-Goals）

- 不追求用 prompt/temperature “逼迫 LLM 精确输出”
- 不把关键决策交给 LLM 自由生成（LLM 可解释，但不做最终决策裁判）
- 不要求把所有指南内容强行树化（避免“假确定性”）
- 不默认让 LLM 直接调用外部系统（HIS/EMR）执行读写（必须通过确定性中间层鉴权与审计）

---

## 4. 核心设计原则

1) **分层（Separation of Concerns）**
- LLM：交互 + 信息抽取 + 解释
- SOP/决策树：决策层（可执行、可测试）
- IM：在线安全门禁（非生成、可审计、可回滚）
- KVI：证据供给与绑定（减少幻觉、支持引用）

2) **编译（Compilation over free-form generation）**
- LLM 输出必须落在结构化 IR（SOP/Plan JSON）内
- IR → 执行计划（Tool Plan）由确定性编译器完成

3) **fail-closed**
- 缺关键变量/缺证据/越出指南适用范围：停止推断并输出“缺口清单”，必要时转人工确认

---

## 5. 系统总体分层（建议形态）

```
User Query
  ├─ LLM 交互层：澄清/追问/抽取（结构化 patient_state）
  ├─ SOP 决策层：基于 patient_state 走决策路径（deterministic）
  ├─ IM 在线门禁：检查证据、适用范围、可执行性、风险控制（fail-closed）
  └─ LLM 解释层：把“决策路径 + 证据引用 + 风险提示”转成可读输出

KVI（证据供给层）
  ├─ 检索：evidence blocks / evidence units
  ├─ 注入：K_ext/V_ext prefix（用于解释与引用的可靠性增强）
  └─ 引用：证据 unit/块 id 可追溯
```

---

## 6. SOP/决策树：作为中间表示（IR），不是“把知识注入 LLM”

要点澄清（避免概念误用）：
- 这套系统并非“把决策树知识注入 LLM 让它更聪明”，而是**把决策权从 LLM 迁移到可审计的 SOP 决策层**。
- LLM 的核心价值变成：**交互（补齐变量）+ 抽取（结构化）+ 解释（可读）**。

---

## 7. 在线推理期流程（端到端）

### 7.1 Step A：按需增量结构化患者状态（降低成本）

1) LLM 先做“最小抽取”：
- 从 query 中抽取已给出的变量（症状/时间/检查/既往史）
- 产出 `patient_state_partial`（字段化，unknown 显式为 unknown）

2) SOP 决策层声明“下一步必需变量”（Just-in-time requirements）：
- 当前节点需要哪些变量才能判定下一分支（例如：血小板计数、出血表现、是否孕妇等）

3) LLM 交互层用最少轮次追问或触发“查询计划”：
- 若接入 HIS/EMR：LLM 只输出“查询意图/字段清单”，由确定性中间层执行查询（权限/审计/脱敏）
- 查询结果回填到 `patient_state`

### 7.2 Step B：SOP 决策推断（deterministic）

- 输入：`patient_state`（结构化）
- 输出：`decision_path`（经过哪些节点、触发哪些条件、得到什么建议）

### 7.3 Step C：IM 在线安全门禁（非生成）

IM 对本次推断做门禁检查：
- **适用范围**：指南版本/人群/场景是否匹配
- **信息充分性**：关键变量是否缺失（缺则必须 stop + 追问/转人工）
- **证据充分性**：关键结论/关键分支是否有 evidence 支持（不足则降级或转人工确认）
- **风险控制**：是否触发高风险步骤（强制 `requires_human_confirmation`）

### 7.4 Step D：KVI 证据供给 + LLM 解释输出

- KVI 从指南 PDF 证据库检索与当前路径相关的 Evidence Units
- LLM 解释层输出：
  - 决策结论（限定为 SOP 的输出，不扩写）
  - 关键分支依据（引用证据）
  - 风险提示与缺口（IM 输出）

---

## 8. SOP 的最小数据协议（建议）

### 8.1 patient_state（结构化患者状态）

- 必须支持 unknown
- 每个字段可携带 provenance（来源：用户描述/EMR 查询/检验单号/时间戳）

### 8.2 sop_node（决策树节点）

- `id`
- `requires[]`：本节点判定必需变量（用于“按需结构化”）
- `conditions[]`：可判定条件（对 patient_state 字段的表达式）
- `actions[]`：建议/下一步（检查/处置/用药建议的“类别化动作”，不是自由文本）
- `evidence_refs[]`：绑定指南条款/证据句（可选但对关键节点建议必填）
- `risk_level` + `requires_human_confirmation`

---

## 9. 指南 PDF → SOP 的半自动编译（降低维护成本）

建议接受的现实：
- 自动化可以做“初稿”，但必须 **人审**（指南存在例外/弱推荐/语义细节）

推荐流程：
1) PDF → Evidence Units（抽取式、可引用）
2) LLM 生成 SOP 初稿（节点/条件/条款引用）
3) 人审修改（差异记录为 patch）
4) 版本化发布（按指南版本）
5) 回归测试：输入样例 patient_state → 期望路径/结论（防漂移）

---

## 10. 与当前仓库/实现的对接点（方向性）

可复用的能力（当前已存在）：
- KVI 检索/注入（KVBank + past_key_values prefix）
- RIM / Gate 的“非生成控制”理念
- PatternContract 的 metadata-only 验证思路
- Evidence Units（sentence-level）与语义路由（simple pipeline 已验证）

需要新增的最小模块：
- SOP schema（IR）+ 校验器
- SOP 决策引擎（deterministic）
- 在线 IM（SOP 专用门禁规则）
- （可选）HIS/EMR 查询中间层（权限、审计、脱敏、白名单字段）
