==============================
Patch A · Slot 判定一致性（最高优先级）
==============================

问题：
- gate.slot_status 与 decision_reason 中的 slot 满足性判定不一致
- 原因：slot 被多次计算，使用了不同的 evidence batch

修改要求：
1. slot_status 的计算 **只允许一次**
2. 所有以下逻辑 **必须使用同一个 slot_status 实例**：
   - decision
   - decision_reason
   - allow_rim_retry
3. slot_status 的输入 evidence 必须来自：
   - filtered / semantic_instances
   - 禁止使用 raw retrieval batch

实现提示（示意）：
- 在 pattern_pipeline 或 runtime 中：
  - 先 build semantic_instances
  - 再 compute_slot_status(semantic_instances)
  - 将 slot_status 作为只读对象向下传递
- 禁止在 gate / decision 阶段重新计算 slot_status

必须新增 debug 日志：
- slot_status_source = "semantic_instances"
- slot_status_snapshot（序列化输出）

==============================
Patch B · LIST_ONLY → Extract-Only（安全关键）
==============================

问题：
- LIST_ONLY 仍允许模型补出 evidence 中不存在的条目
- 导致“无根据列表生成”

修改要求：
1. 当 final_answer_style == LIST_ONLY 时：
   - 禁止任何自由生成
   - 只允许从 slot.evidence_blocks 中抽取
2. 每一个 list item 必须满足：
   - 可追溯到 evidence_block_id
   - 来自 evidence 的显式 span / bullet / enumeration
3. 如果 slot 满足但无法抽取 ≥1 item：
   - 强制 decision = REFUSE
   - decision_reason = "slot-satisfied-but-not-extractable"

实现提示（示意）：
- 新增 extract_list_items(evidence_blocks) 函数
- LIST_ONLY 路径下：
  - 禁止调用 LLM 自由生成
  - 仅格式化 extract_list_items 的返回结果

必须新增 debug 日志：
- list_items_extracted (array)
- list_items_source_block_ids
- list_extraction_failed_reason（如为空）

==============================
禁止事项（必须遵守）
==============================

- 不允许在 LIST_ONLY 下出现 schema 推导出的新概念
- 不允许出现 evidence 中不存在的术语
- 不允许 fallback 到 baseline_answer

验收标准

3) 若 evidence 为统计分析/方法学段落：
   - LIST_ONLY 必须 REFUSE，而不是“猜一个列表”
